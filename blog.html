Week 4

April 21, 2016

Currently for KeyboARd, we’re trying to make sure our application will work as we intended. We’ve decided to not buy a keyboard model, but instead make one ourselves.

A keyboard is essentially multiple white rectangles with black rectangles stacked on top of them. By creating the model ourselves we allow easier piano customization if needed as opposed to trying to understand how to add on to other people’s models.

Currently, the team is in investigation mode to ensure that we are able to create KeyboARd successfully despite possible restrictions caused by the Hololens. So far so good, we feel that we should be able to place our holo-piano on our physical piano and the restricted vision shouldn’t be a major problem.

Next week, our goal is to integrate our new MIDI keyboard to Unity and have our model keyboard light up when we hit the corresponding key. We also have to create some sort of representation for our music so our users can play a song.

What we are doing: MIDI integration with keyboard and create a simple song to test with

  • Josh: Work with Keyboard MIDI integration
  • Seokmin: Work with Keyboard MIDI integration
  • Hayoun: Explore reading MIDI files for songs
  • Matt: Create simple song and work on lighting up keyboard

Week 5

April 27, 2016

This was a big week for us in the holographic world. Looking back to a week ago, all we had was a static piano model that we made, a keyboard that didn’t do anything yet, and high hopes. However, this week we made a ton of progress. As of now, a simple song (Mary had a Little Lamb) lights up the keys and the notes rain in from the distance.

We also were able to connect the physical keyboard to the hololens via the networking tools in unity and get them to communicate with one another. Now when a user opens the app, they can press keys on the physical keyboard and receive visual feedback on whether or not the right note is being played based on the color that comes up. Green for correct key pressed and red for incorrect key pressed.

A huge victory for the week was today when Hayoun figured out how to read in a MIDI file that we found online, parse it and extract the notes that were supposed to be played, and make it successfully light up the keyboard playing Beethoven’s Fur Elise.

This is a huge opportunity for us to have hundreds or thousands of songs available. Instead of hard coding songs, with a little more work we can download a midi file from anywhere and have it work with our system.

Our goals for the rest of this week and next week include:

  • Positioning virtual keyboard on top of actual keyboard.
  • Determine best way to handle more advanced songs.
    • Right now we can only work with songs that play one note at a time
  • Continue work on MIDI reading.
    • Fixing minor glitches in generating KeyColor
    • Integrating MusicRain and the MIDI reader
  • Create Simple menu.
    • Select Song (If all goes well with MIDI)
    • Start button
  • Ability to restart song.
    • Currently we have to re-launch the application in order to restart
  • Key press feedback accuracy and effects.

We're looking forward to seeing what comes this next week and how well we can figure out these issues!

Week 6

May 5, 2016

We had another awesome week getting some pretty big things figured out for our project. Most of the week we were stuck on how to corrrectly move the keyboard using the TapToPlace script from the HoloToolkit but on Tuesday we solved that issue, which of course presented us with a new issue. When the keyboard was placed, it seemed to make the music rain from the original spot of the keyboard and move toward the new spot which looked very bad and was pretty much unusable. However, in the last 30 minutes of lab, our team found a solution using some intense trigonometry. Essesntially when we rotated the keyboard in whatever way, we needed to make sure that the distance from the keys to where the rain started was 1 meter. That was the hypotenuse of our triangle and using the angle of the rotation in the Y axis, we were able to adjust our x and z position transforms to correctly handle this rotation.

We also made great progress in to reading in a MIDI file and if that can work out, we will have access to any song we can find a file for to play on the piano.

For next week, our goal is to incorporate voice commands, allow for a manual positioning of the keyboard using some of the keys to incrementally move the keyboard to the right spot, and if there is time, to use a wav file for each of the notes which can be played though the hololens speakers when a note is pressed, and figure out a better way to show a user a note is coming and get ready to press it.

To do this: Seokmin and Matt will begin work on voice commands, Josh will work on manually placing the keyboard, and Hayoun will finish up the MIDI work and start on finding wav files for the sounds.

We are very pleased with our progress so far and can't wait to get working on the next part of the project!

Week 9 KeyboARd Update

May 19, 2016

After all the troubles we had last week with our repository not working and having to basically recreate that, we actually ended up having a very productive week. We implemented more manual adjustment controls for the keyboard, made the keyboard fade away once it’s placed so all you see are the lights, fixed an issue with the timing of the key light up, created a main menu UI, downloaded some more simple songs, created a tutorial to teach users how to use the app, and are working towards making the keyboard placement happen only one time and having that position persist through every scene so the user never has to worry about picking up and placing the keyboard themselves.

The toughest issue that we faced this week was the timing of the key light up. The problem arose because with the midi files, not all notes are the same length and because of that, the music rain would come out of the front of the keyboard and the note would stop lighting up too early. We figured out that because the position of the notes are measured from their centers, using some math would allow us to determine if the top of the music rain was at the top of the key (signifying that the note was done). To solve the issue of the music rain running through the front of the keyboard, we created a “cover” that was completely black and all of the notes ran out of the keyboard under it. So that when someone is using it in the hololens, the cover looks transparent but all the notes below it are occluded.

This week will be our time to double down our efforts in order to make the dress rehearsal demo look polished. Our goal is to make the UI cleaner and simpler, use a hand model for our tutorial, focus really hard on figuring out position persisting across multiple scenes so that the piano only has to be set once, and lots of play testing.

Josh will be working on the UI and connecting all of the scenes together in a clean manner. Seokmin is working on the position persisting of the virtual keyboard. Hayoun is working with the hand model to help in the tutorial. And Matt is putting the finishing touches on the tutorial, starting the final video, and beginning work on the UI as well.

We hope that the demo next week goes very well and we expect it to, but it will also be good for us to get a lot of eyes on our project so that we can have everything perfectly polished when it comes time for our real demo on the 9th.

Week 10 KeyboARd Update

June 3, 2016

From the user testing experience we had with our demo. We've decided to make some changes:

  1. Without any guidance from us, users didn't really know how to look down to the keyboard from the menu and vice versa when performing the next steps.
    • To fix this, we’re going to create arrows that will show the users where they have to look
  2. Having to adjust the virtual keyboard every time we run into problems takes up a lot of time.
    • Luckily, one of the HoloLens developers told us about Anchor Points which will keep our keyboard positioned in the same location. We hope by integrating anchor points we will have a more consistent experience
  3. We need more keyboards. Currently we can only have users test one at a time
    • We will get another keyboard to allow us to have 2 people try our app at a time. To do this we need
      • 3 computers. 2 will be a server to detect keyboard hits and 1 to display our video.
      • We will need 2 Ethernet cables for 2 of our computers
      • 4 HoloLens, 2 for use and 2 for backup
      • Another keyboard stand
    • We’re also considering setting up a sign in sheet to reserve spots so that people who are waiting can look around and we can call them back when it’s their turn
  4. Stereo System to attract more people
    • One thing people have always asked us if we had any sound not knowing that only the users will hear the sound. We intend to make it more fun (and attract people) by broadcasting what everyone is playing on a stereo
  5. Integrate a score system with a white board leaderboard to make games competitive for the user and to drive more traffic to our app
    • We’ll make a score system that will track how many notes that the user hits correctly and we’ll have a high score white board that we will have to keep everyone’s score
  6. Fix a bug with the scale when the user plays 2 notes quickly so that they are played almost at the same time that the app won’t continue anymore

Overall the demo was a fun learning experience and we have a lot of work ahead of us to attract more people and create a smoother and fun game experience!
